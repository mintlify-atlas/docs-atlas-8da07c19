Turbo Query is an open-source distributed hybrid search engine. We welcome contributions of all kinds, from bug fixes to new features.

## Project Structure

The codebase is organized into three main directories:

```
turbo-query/
├── cmd/
│   ├── api/            # Coordinator server (fan-out layer)
│   └── shard/          # Per-shard search server
├── internal/
│   ├── embed/          # Embedding client + normalization
│   ├── server/         # Coordinator HTTP handlers
│   ├── shard/          # Offline indexing pipeline
│   └── shardnode/      # Shard HTTP handlers and hybrid search
└── data/               # Generated shard indexes and vector files
```

### Directory Details

<AccordionGroup>
  <Accordion title="cmd/ - Entry Points">
    Contains the main packages for running services:
    
    - `cmd/api/main.go` - Coordinator server that will fan out queries (in progress)
    - `cmd/shard/main.go` - Individual shard server for hybrid search
  </Accordion>
  
  <Accordion title="internal/embed/ - Embeddings">
    Handles text embedding generation via Ollama:
    
    - `Embed(text string)` - Generate MiniLM embeddings
    - `normalize(vec []float32)` - L2-normalize vectors for cosine similarity
    - Vector dimension: **384** (from `all-minilm` model)
  </Accordion>
  
  <Accordion title="internal/shard/ - Indexing">
    Offline indexing pipeline implementation:
    
    - `ingestWiki()` - Stream JSON documents
    - `worker()` - Generate embeddings in parallel
    - `router()` - Consistent hash distribution
    - `shardWriter()` - Batch index to Bleve
    - `initShardStorage()` - Create mmap vector files
  </Accordion>
  
  <Accordion title="internal/shardnode/ - Query">
    Per-shard search and hybrid ranking:
    
    - BM25 candidate retrieval via Bleve
    - Vector similarity reranking
    - Score normalization and fusion
    - HTTP search endpoint
  </Accordion>
</AccordionGroup>

## Getting Started

<Steps>
  <Step title="Clone the repository">
    ```bash
    git clone https://github.com/abhiram-karanth-core/turbo-query
    cd turbo-query
    ```
  </Step>
  
  <Step title="Install dependencies">
    ```bash
    go mod download
    ```
  </Step>
  
  <Step title="Start Ollama (for embeddings)">
    ```bash
    docker-compose up -d ollama
    ```
    
    Pull the MiniLM model:
    ```bash
    docker exec -it ollama ollama pull all-minilm
    ```
  </Step>
  
  <Step title="Build the project">
    ```bash
    make build
    ```
    
    This compiles the API server to `main.exe`.
  </Step>
</Steps>

## Development Workflow

### Live Reload with Air

For rapid development, use [Air](https://github.com/air-verse/air) for live reloading:

```bash
make watch
```

Air configuration (`.air.toml:5-28`):

```toml
[build]
  args_bin = []
  bin = ".\\main.exe"
  cmd = "make build"
  delay = 1000
  exclude_dir = ["assets", "tmp", "vendor", "testdata", "node_modules"]
  exclude_file = []
  exclude_regex = ["_test.go"]
  exclude_unchanged = false
  follow_symlink = false
  full_bin = ""
  include_dir = []
  include_ext = ["go", "tpl", "tmpl", "html"]
  include_file = []
  kill_delay = "0s"
  log = "build-errors.log"
  poll = false
  poll_interval = 0
  post_cmd = []
  pre_cmd = []
  rerun = false
  rerun_delay = 500
  send_interrupt = false
  stop_on_error = false
```

## Code Organization

### Adding New Features

<Steps>
  <Step title="Identify the layer">
    Determine if your feature belongs in:
    - `cmd/` - New service binary
    - `internal/` - Internal package
    - Update existing handlers
  </Step>
  
  <Step title="Write tests first">
    Create `*_test.go` files following Go conventions:
    ```go
    package mypackage
    
    import "testing"
    
    func TestMyFeature(t *testing.T) {
        // Test implementation
    }
    ```
  </Step>
  
  <Step title="Implement the feature">
    Follow existing patterns in the codebase:
    - Use meaningful variable names
    - Add comments for complex logic
    - Keep functions focused and small
  </Step>
  
  <Step title="Run tests and build">
    ```bash

    make build
    ```
  </Step>
</Steps>

## Debugging

### Common Issues

<AccordionGroup>
  <Accordion title="Embedding service not available">
    **Error:** `embed http error: connection refused`
    
    **Solution:** Ensure Ollama is running:
    ```bash
    docker-compose up -d ollama
    docker exec -it ollama ollama pull all-minilm
    ```
  </Accordion>
  
  <Accordion title="Bleve index corruption">
    **Error:** `index error: ...`
    
    **Solution:** Delete and rebuild the index:
    ```bash
    rm -rf data/shard-*/index.bleve
    # Re-run the indexing pipeline
    ```
  </Accordion>
  
  <Accordion title="Out of memory during indexing">
    **Error:** `fatal error: out of memory`
    
    **Solution:** Reduce batch size in `internal/shard/shard.go`:
    ```go
    const batchSize = 50  // Reduced from 100
    ```
  </Accordion>
</AccordionGroup>

## Submitting Changes

<Steps>
  <Step title="Create a feature branch">
    ```bash
    git checkout -b feature/your-feature-name
    ```
  </Step>
  
  <Step title="Make your changes">
    Follow the code style and testing guidelines above.
  </Step>
  
  <Step title="Commit with clear messages">
    ```bash
    git add .
    git commit -m "feat: add distributed query coordinator"
    ```
    
    Use conventional commit format:
    - `feat:` New features
    - `fix:` Bug fixes
    - `docs:` Documentation
    - `refactor:` Code refactoring
    - `test:` Test additions
  </Step>
  
  <Step title="Push and open a PR">
    ```bash
    git push origin feature/your-feature-name
    ```
    
    Open a pull request on [GitHub](https://github.com/abhiram-karanth-core/turbo-query)
  </Step>
</Steps>
