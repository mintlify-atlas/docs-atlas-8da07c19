# Quickstart Guide

Get Turbo Query running locally in minutes and execute your first hybrid search query.

## Prerequisites

Before starting, ensure you have:

- **Docker** and **Docker Compose** installed
- **curl** or any HTTP client for testing
- At least **4GB** of available RAM (for 4 shards + Ollama)

## Launch the System

<Steps>
  <Step title="Clone the repository">
    ```bash
    git clone https://github.com/abhiram-karanth-core/turbo-query.git
    cd turbo-query
    ```
  </Step>

  <Step title="Start all services with Docker Compose">
    This will launch 4 shard servers and the Ollama embedding service:

    ```bash
    docker-compose up -d
    ```

    <Note>
    The `docker-compose.yml` file starts:
    - **4 shard servers** on ports 9001-9004 (mapped from internal port 8080)
    - **Ollama** on port 11434 for MiniLM embeddings
    </Note>

    Each shard server runs with these environment variables:
    - `PORT=8080` - Internal HTTP server port (source/docker-compose.yml:6)
    - `SHARD_ID=0,1,2,3` - Unique shard identifier (source/docker-compose.yml:7)
  </Step>

  <Step title="Verify services are running">
    Check that all containers are healthy:

    ```bash
    docker-compose ps
    ```

    You should see:
    - `shard0` (port 9001)
    - `shard1` (port 9002)
    - `shard2` (port 9003)
    - `shard3` (port 9004)
    - `ollama` (port 11434)

    Test a shard health endpoint:

    ```bash
    curl http://localhost:9001/health
    ```
  </Step>

  <Step title="Execute your first search query">
    Send a search request to any shard server. Here's an example searching shard0:

    <CodeGroup>
    ```bash curl
    curl -X POST http://localhost:9001/search \
      -H "Content-Type: application/json" \
      -d '{
        "query": "distributed search engine",
        "top_k": 10
      }'
    ```

    ```python Python
    import requests

    response = requests.post(
        "http://localhost:9001/search",
        json={
            "query": "distributed search engine",
            "top_k": 10
        }
    )

    results = response.json()
    print(results)
    ```

    ```javascript JavaScript
    const response = await fetch('http://localhost:9001/search', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        query: 'distributed search engine',
        top_k: 10
      })
    });

    const results = await response.json();
    console.log(results);
    ```
    </CodeGroup>
  </Step>
</Steps>

## Understanding the Request

The search endpoint accepts a JSON payload with two fields:

```json
{
  "query": "your search text",
  "top_k": 10
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `query` | string | Yes | The search query text (source/internal/shardnode/types.go:4) |
| `top_k` | integer | No | Number of results to return (default: 10) (source/internal/shardnode/types.go:5) |

## Understanding the Response

The API returns a JSON object containing an array of search hits:

```json
{
  "hits": [
    {
      "doc_id": "42",
      "score": 0.8734,
      "shard_id": "0",
      "title": "Introduction to Distributed Systems",
      "text": "A distributed system is a collection of independent computers..."
    }
  ]
}
```

Each hit includes (source/internal/shardnode/types.go:8-14):

| Field | Type | Description |
|-------|------|-------------|
| `doc_id` | string | Document identifier within the shard |
| `score` | float64 | Hybrid relevance score (0.7 × BM25 + 0.3 × cosine) |
| `shard_id` | string | Which shard returned this result |
| `title` | string | Document title |
| `text` | string | Document content |

<Info>
The hybrid score combines:
- **70% normalized BM25** for keyword relevance (source/internal/shardnode/routes.go:113)
- **30% normalized cosine similarity** for semantic relevance
</Info>

## Query Flow

When you send a search request, here's what happens:

1. **Query Embedding** (source/internal/shardnode/routes.go:71)
   - The query text is sent to Ollama for MiniLM embedding
   - Returns a 384-dimensional normalized vector

2. **BM25 Retrieval** (source/internal/shardnode/routes.go:77-85)
   - Bleve performs full-text keyword search
   - Retrieves top 100 candidates for reranking

3. **Vector Reranking** (source/internal/shardnode/routes.go:99-114)
   - For each candidate, fetch its vector from the mmap store
   - Compute cosine similarity via dot product
   - Calculate hybrid score: `0.7 × BM25_norm + 0.3 × cosine_norm`

4. **Top-K Selection** (source/internal/shardnode/routes.go:132-138)
   - Sort by hybrid score descending
   - Return top K results with full metadata

<Tip>
Typical per-shard latency is **10-15ms** for local queries. The embedding generation step dominates the total query time.
</Tip>

## Testing Different Shards

You can query individual shards directly to understand data distribution:

<CodeGroup>
```bash Shard 0 (port 9001)
curl -X POST http://localhost:9001/search \
  -H "Content-Type: application/json" \
  -d '{"query": "machine learning", "top_k": 5}'
```

```bash Shard 1 (port 9002)
curl -X POST http://localhost:9002/search \
  -H "Content-Type: application/json" \
  -d '{"query": "machine learning", "top_k": 5}'
```

```bash Shard 2 (port 9003)
curl -X POST http://localhost:9003/search \
  -H "Content-Type: application/json" \
  -d '{"query": "machine learning", "top_k": 5}'
```

```bash Shard 3 (port 9004)
curl -X POST http://localhost:9004/search \
  -H "Content-Type: application/json" \
  -d '{"query": "machine learning", "top_k": 5}'
```
</CodeGroup>

<Note>
Documents are distributed across shards using consistent hashing. Different shards will return different documents for the same query.
</Note>

## Viewing Logs

Monitor shard performance and query latency:

```bash
# All services
docker-compose logs -f

# Specific shard
docker-compose logs -f shard0

# See latency metrics
docker-compose logs shard0 | grep latency
```

Example log output (source/internal/shardnode/routes.go:56-59):
```
shard=0 latency=12.4ms
received search: distributed search engine
```

## Stopping the System

When you're done:

```bash
# Stop all containers
docker-compose down

# Stop and remove volumes
docker-compose down -v
```
