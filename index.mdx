# Welcome to Turbo Query

Turbo Query is a high-performance distributed search system that combines BM25 keyword retrieval with semantic vector search using MiniLM embeddings. Built with production-grade infrastructure concepts, it demonstrates advanced search engine architecture including sharding, fan-out querying, and memory-mapped vector storage.

## What is Turbo Query?

Turbo Query is a distributed hybrid search engine that combines the best of both worlds:

- **Keyword Search (BM25)**: Fast, precise full-text retrieval using inverted indexes
- **Semantic Search (Vector)**: Context-aware similarity matching using MiniLM embeddings
- **Hybrid Ranking**: Intelligent fusion of both approaches for superior relevance

The system uses consistent hash-based sharding to distribute documents across multiple shard servers, each maintaining its own Bleve inverted index and memory-mapped vector store for efficient zero-copy reads.

## How It Works

Each search query flows through the following pipeline:

1. **Query Embedding**: The search query is converted into a 384-dimensional vector using MiniLM
2. **BM25 Retrieval**: Each shard retrieves top keyword matches using Bleve full-text search
3. **Vector Reranking**: Candidates are rescored using cosine similarity with the query embedding
4. **Hybrid Fusion**: Final scores combine normalized BM25 and semantic scores (70% BM25 + 30% semantic)
5. **Top-K Results**: The highest-scoring documents are returned with full metadata

<Info>
Typical per-shard latency is **10-15ms** for local queries, with embedding generation being the primary cost factor.
</Info>

## Scoring Strategy

Turbo Query uses a weighted hybrid scoring formula:

```text
final_score = 0.7 × normalized_BM25 + 0.3 × normalized_cosine
```

This balances:
- **Precision** from exact keyword matching (BM25)
- **Recall** from semantic understanding (vector similarity)

## Architecture Overview

```text
User Query
    ↓
Coordinator (fan-out)
    ↓
Shard Servers (parallel)
    ↓
BM25 retrieve → Vector rerank → Hybrid score
    ↓
Top-K merge → Response
```

Each shard maintains:
- A **Bleve** inverted index for keyword retrieval (source/internal/shardnode/server.go:50)
- A **memory-mapped** dense vector store at `/data/vectors.bin` (source/internal/shardnode/server.go:47)
- A shard-local document ID mapping in `/data/docmap.json` (source/internal/shardnode/server.go:48)
- An HTTP search endpoint on port 8080 (source/docker-compose.yml:6)
